# SRT → Character Sheet + Scene Image Prompts

Turn one timestamped SRT script into two text files:

1. **`<name>_characters.txt`** — a numbered character sheet: one fixed, full visual description per character. This is the **locked reference source** (use it to make a reference image, and to keep wording stable). It is NOT pasted verbatim into the scene prompts.
2. **`<name>_scenes.txt`** — a numbered list of scene image-prompts, **exactly one per SRT block**, in order. Each prompt is **self-contained**: it names the present characters (name + scene garment, plus an age phrase only for flashbacks), states the action, the camera angle, and **re-describes the physical setting and lighting in full**, ending with a scene-appropriate lighting clause + `pro color grading` + the locked style string.

Two ideas drive everything:
- **Character recognizability** — a character must read as the same person in every prompt. In prompts this is carried by the **`@tag` + the locked per-act garment** (the full physical description lives on the character sheet, whose multi-view reference-sheet prompt generates the reference image the tag loads). Never let a character's identity drift.
- **1:1 mapping** — block N of the SRT → prompt N of the scenes file. One voiceover clip = one illustration.

**Both files are derived from an internal JSON scene-layout map built FIRST.** Before writing either deliverable, you map the whole story into a JSON object (Step 2) — `storyContext`, a `characters[]` array with each character's locked description, and a `sceneMap[]` locking every scene's concrete location, time, lighting, cast, positions, outfits, ages-per-time-anchor, and lead subject — cross-referenced line by line against the SRT. The character sheet and the scene prompts both read from this map, so the look, wardrobe, lighting, and blocking are decided once and stay consistent. The JSON is internal: it is never saved or shown to the user.

**The image tool has NO memory between prompts.** It cannot see prompt N−1 when it renders prompt N. So every prompt must spell out its own world: the room and what is in it, the surroundings, what each person is wearing, and the light. Anything referred to only by a label ("the same office", "the Voss Road living room") or by comparison ("a similar kitchen", "as before") will not be understood. Describe it concretely, every time. (The one exception is the `@tag` for a character, which the user has pre-loaded as a reference image.)

## Workflow (do these in order)

### Step 0 — Read the reference files first
Before writing anything, read all five reference files. They contain the exact formulas that make the output match the user's house style:
- `references/character-formula.md` — how to write the **character-sheet** description (attribute order, the eye-"quality" pattern, fabric/garment vocabulary), how to build each character's **`@tag`** and lead each sheet line with it (in place of the name), AND how a character is referred to inside a scene prompt (`@tag` + garment, + age phrase for flashbacks).
- `references/scene-layout-map.md` — how to build the **internal JSON scene-layout map** (a planning artifact, not a delivered file) that the whole job is mapped into FIRST: `storyContext`, a `characters[]` array with each character's locked `suggestedDescription`, and a `sceneMap[]` locking each scene's concrete `location`, `timeOfDay`, `lighting`, `charactersPresent`, `characterPositions`, `sceneOutfits`, `characterAges` (per time anchor), `leadSubject`, `inFrame`/`mentioned`, before any character sheet or prompt is written.
- `references/scene-prompt-formula.md` — how to assemble one self-contained scene prompt, including the no-dialogue rule, the each-character-named-once rule, and how to handle object-only / single-character shots.
- `references/camera-and-style.md` — the camera-angle rotation pool (worded to avoid naming a character twice), the location formula (a **described** place, not a bare label), the per-scene lighting clause pool, the `pro color grading` note, and the **locked style string** that ends every scene prompt.
- `references/minor-and-policy-safety.md` — **MANDATORY** rules for any script with children, real public-figure names, or brands. The image tool runs a non-configurable safety filter that silently rejects an image it reads as a minor interacting with an unrelated adult (and real celebrity names / brand logos). The rule is that **children are ALWAYS kept in every frame the story places them in** — never pulled or substituted. Most child scenes (a child alone, with their own family, or in a crowd) render as written; a child sharing a frame with a non-parent adult is KEPT and the composition is reworked (parent anchored, gaze/contact redirected through the parent or a prop) so it passes the filter. Read it whenever the cast includes a child, a named real person, or a branded product, and apply it in Steps 2 and 4.

### Step 1 — Parse the SRT
Read the SRT. Each block = `index`, `start --> end` timestamp, and text. Keep the block index; the scenes file must have the same number of entries in the same order. Do not merge or split blocks — one block, one prompt. (If a block is pure narration with no concrete action, you still produce one prompt; pick a fitting visual or metaphor — see scene formula.)

### Step 2 — Build the internal JSON scene-layout map (do NOT skip, do NOT deliver)
This is the planning step that the whole job is mapped into **first**, before any character sheet or prompt. Read the **entire** script, cross-reference it line by line against the source story (the most recent full story in the conversation, if one exists; otherwise the SRT is the only source), and build the JSON object defined in `references/scene-layout-map.md`:
- the top-level `language` / `storyContext` / `mainSetting` / `storyType`,
- a `characters[]` array listing every named character plus recurring unnamed roles (e.g. "the Uniformed Officer", "the Valet Supervisor"), each with its `name`, `role`, `firstAppearanceLine`, `tag` (derived per `references/character-formula.md`, e.g. `@denzel`), and a complete `suggestedDescription` (the locked canonical look),
- a `sceneMap[]` that divides the whole SRT into scenes — runs of consecutive blocks sharing one location and one time anchor — locking each scene's concrete `location`, `timeOfDay`, `lighting`, `sceneSummary`, `charactersPresent`, `characterPositions`, `sceneOutfits`, `characterAges` (filled ONLY when the scene's time anchor differs from a character's first appearance), `leadSubject`, `inFrame`, `mentioned`, `dramaticRelationship`, and `crowdNotes`.

Every field must be traceable to the story/SRT — do not invent or omit characters, locations, or events. **Scene boundaries follow the story's logic: a change in location OR a change in time anchor = a new scene.** The scenes in order must cover the entire SRT with no gaps and no overlaps; the sum of all `startLine`–`endLine` spans must equal the SRT block count.

Keep this JSON for your own use only — it is **not** saved and **not** shown to the user. The deliverables are still just the two `.txt` files. The JSON exists so that when you write the character sheet and the prompts you are reading locked facts (each character's look, each scene's concrete location, lighting, cast, positions, wardrobe, lead subject), not re-deciding them block by block. Write it in English even when the SRT is in another language.

**While building the map, sort each child scene into a bucket.** For any scene whose cast includes a child (or any real public-figure name / brand), note which bucket it is, per `references/minor-and-policy-safety.md` §A: **GREEN** (child alone, with their own family, or in a crowd — keep the child, write it naturally) or **YELLOW** (child shares a frame with a non-parent adult — keep the child and compose safely via the §B recipe: parent in frame, child anchored, every adult on a separate task, gaze/contact redirected through the parent or a prop). **The child is ALWAYS kept in frame — there is no "remove the child" bucket.** Decide the composition once here so you are not re-litigating it block by block.

### Step 3 — Build the character sheet (the locked source), derived from the JSON map
Using the `characters[]` array from the JSON map, write the `<name>_characters.txt` file: for each character, ONE fixed, full description following `references/character-formula.md` — a **multi-view reference-sheet prompt that leads with the character's `@tag` (in place of their name)**, built from that character's `suggestedDescription`. The `@tag` (already assigned in the map, e.g. `@denzel`) opens each sheet line so the reference image it generates can be saved directly under that tag — the same tag the scene prompts call — with no renaming step. The sheet is the canonical look and the source of truth for appearance.

Rules:
- One canonical identity per character. Do **not** drift (same age, skin tone, build every time). If the script is contradictory, pick one reading and keep it — the same reading already locked in the JSON map's `suggestedDescription`.
- Write entirely in **English**, even if the SRT is in another language.
- Infer ethnicity/age/build/clothing entirely from the script. Make no default assumptions about era, place, or ethnicity — derive the world from the SRT's own content, and if the script implies another time period or setting, adapt fully to it.
- Output format = the character-sheet line format in `references/character-formula.md` (multi-view full-body framing + neutral expression + the character-reference-sheet style tail).

### Step 4 — Generate scene prompts, one per block, scene by scene
Work through the JSON map's `sceneMap[]` **one scene at a time, in order** — do not batch the whole script in a single pass, because that is what causes wardrobe/lighting/cast/setting drift. For each scene, re-read its JSON entry first (concrete `location`, `lighting` clause, `inFrame` cast, that scene's `sceneOutfits`, `characterPositions`, `characterAges`, `leadSubject`, `dramaticRelationship`). Then write one prompt for each block in that scene's `startLine`–`endLine` span, following `references/scene-prompt-formula.md`:
- Identify who/what is in frame for that block's specific text (a subset of the scene's `inFrame` list is fine; never add someone the scene's `charactersPresent` didn't account for, and never draw anyone who is only in `mentioned`).
- The scene's `leadSubject` is the primary subject of the block unless that block's text clearly shows someone else performing the main action — cross-reference each line, do not default to the protagonist.
- Write each present character as their **`@tag`** + `wearing <that scene's `sceneOutfits` garment>`, and — **only if `characterAges` is filled for this scene** (a flashback / younger or older version) — the age phrase from that field (e.g. "young @denzel as a twelve-year-old boy", "a younger @marcus in his 30s"). Do NOT add ethnicity/build/skin/hair/eyes in the prompt; that lives on the sheet. **If the shot is full-body / wide (legs and feet visible), the scene garment must also name the lower garment + footwear** so the character isn't rendered barefoot (Hard rule 13); upper-body framings need only the top garment.
- **Use each `@tag` at most ONCE in the whole prompt.** Tagging the same person twice makes the image tool render a duplicate of them. If the camera phrase refers to the subject, use a pronoun ("seen from over his shoulder"), not the tag again.
- **No dialogue.** Never include spoken words or quotation marks. Render the *act* of speaking as a visible expression/gesture instead ("speaking with a calm, resolved expression"), never the words said.
- **Re-describe the setting in full.** Fold the scene's concrete `location` description into `located in <described place> during <time/light>`. Never use a bare place label, "the same …", "a similar …", or "as before".
- Give **every present character** (the primary subject and each secondary one) a specific, visible emotion — face **plus** posture/gesture — drawn from the block's emotional beat and consistent with both the scene's `dramaticRelationship` and that character's sheet eye-"quality". Never leave anyone in a flat, neutral, lifeless pose ("standing by the desk", "holding a folder"); always say what they feel. Two characters in a frame usually feel different things. **Exception: children get a milder sub-range (curiosity, shyness, quiet worry, delight, a hopeful/uncertain look) — never a dark/intense descriptor (`terror`, `anguish`, `despair`, `grief`, `rage`, `trauma`); move that intensity onto a nearby adult instead.** See Hard rule 12 and `references/scene-prompt-formula.md`.
- **Apply the minor/policy-safety check (`references/minor-and-policy-safety.md`) to every block. Children are ALWAYS kept in every frame the story places them in — never pulled or substituted.** GREEN blocks (child alone, with their own family, or in a crowd) — write naturally; a parent and child can interact warmly with each other. YELLOW blocks (child + a non-parent adult, including beats whose point is that adult) — keep the child and compose safely: parent in frame, child anchored (`beside her`), every adult on a separate task, no unrelated-adult gaze/approach/contact toward the child, the unrelated adult's focus and any contact redirected through the parent or a prop (gift in the parent's hands, stuffed rabbit the child holds), child reacting on their own. If the parent is missing from the source for that beat, ADD the parent to anchor the child — never remove the child. Always keep children wholesome: awake, upright, clothed for the era, supervised — never `half asleep`/lying down/undressed/alone near a vehicle (wholesomeness rule, even with a parent present). **Also never place a child near a physical hazard (fire, a weapon, a height, water, a moving vehicle) — if the source implies danger, depict the moment of safety instead (§F).** Reword §C filter-tripping phrasings (unrelated-adult contact, kneeling to a child, unrelated-adult↔child gaze, low-angle-up, split-narrative, reassurance words) so the child stays in frame. Genericize real brand names/logos, and treat any real public-figure name as a known block risk (flag to the user; fall back to a non-celebrity tag only if a prompt stays blocked after the safety rewrites).
- Choose a camera angle, **rotating** through the pool in `references/camera-and-style.md` so consecutive prompts don't repeat the same framing.
- End with the scene's single locked lighting clause (which must agree with the `during …` cue), then `, pro color grading.`, then the **locked style string**. Never put "bright clear lighting" on a night, evening, or stormy scene.

A scene spanning blocks 30–44 must produce exactly 15 prompts (44 − 30 + 1). Number every prompt to match its SRT block. Finish one scene, then move to the next, until every block has a prompt.

### Step 5 — Write the two files and present them
Save both `.txt` files to `/mnt/user-data/outputs/` and present them. Before finishing, verify the scene file's entry count equals the SRT block count (it must equal the sum of all scene spans in the JSON map).

## Hard rules (these are what "quality" means here)

1. **Count match**: number of scene prompts == number of SRT blocks. Always verify before finishing.
2. **Self-contained prompts**: every scene prompt must describe its own setting, surroundings, wardrobe and lighting from scratch. The image tool cannot see other prompts. Forbid bare location labels with no description, and forbid cross-prompt references like "the same office", "a similar kitchen", "as before", "the previous room". If a location recurs, re-describe the **same concrete physical features** each time so it reads as one place.
3. **No dialogue / no spoken words**: scene prompts never contain quotation marks or quoted speech. Convert any line of dialogue into a visible expression or gesture ("speaking quietly with a weary expression"). Combined with the locked style string, the image must carry **no text and no subtitles**.
4. **Each `@tag` used at most once per prompt**: never repeat a character's tag within a single prompt — it causes the image tool to duplicate the person. Camera phrases referring to the subject use a pronoun, not the tag. A second person's tag is used once, where they first appear in the prompt.
5. **`@tag` + garment carry identity in prompts**: scene prompts refer to a character by their **`@tag`** (exactly as on the sheet, e.g. `@denzel`) + `wearing <scene garment>`, plus an age phrase only for flashbacks. They do NOT use the full name and do NOT repeat the sheet's physical attributes (ethnicity, age band, build, skin, hair, eyes, facial hair). Truly anonymous, non-sheet background figures use short descriptive phrases ("a few diners", "a young man in a beat-up car") and get no tag.
6. **Map first, then generate sequentially**: always build the internal JSON scene-layout map (Step 2) before writing the character sheet or any prompt, and generate prompts one scene at a time in order (Step 4), re-reading each scene's locked concrete `location`, `lighting`, `inFrame` cast, `characterPositions`, and `sceneOutfits`. Never batch the whole script in one pass — that is the main cause of drift. The JSON map is internal; it is never one of the delivered files.
7. **One canonical identity** per character across the whole video (defined on the sheet: ethnicity, age, build, skin, hair, eyes+quality, facial hair). Only the garment changes between scenes (wardrobe-by-act), plus an explicit age phrase for flashbacks.
8. **All English**, no stray foreign words — in both the prompts and the internal JSON map.
9. **Scene-appropriate lighting + `pro color grading` + locked style string**: every scene prompt ends with a lighting clause that fits its time/weather, then `, pro color grading. Rendered in the style of Cinematic realistic 35mm. No text, no subtitles.` The lighting clause varies; everything from `pro color grading.` onward is byte-for-byte fixed. Lighting must agree with the `during …` cue (see `references/camera-and-style.md`).
10. **Camera rotation**: vary framing across consecutive prompts; never the same angle three times running unless the action genuinely continues.
11. **Minor & policy safety (children are ALWAYS kept in frame, composed safely)**: every child the story places in a scene appears in that scene's prompt — always, no exceptions. We never pull, drop, or substitute a child out of a frame. The image tool's filter only fires on one narrow pattern — an unrelated adult interacting with a child (gaze, approach, kneeling, touch) with no parent anchoring them — so we compose around it. GREEN scenes (child alone, with their own family, or in a crowd) render naturally — write them. YELLOW scenes (child + a non-parent adult, including beats whose point is that adult) keep the child via the supervised recipe — parent in frame, child anchored (`beside her`), every adult on a separate task, no unrelated-adult gaze/approach/contact toward the child, and the adult's focus + any contact redirected through the parent or a prop (gift in the parent's hands, stuffed rabbit the child holds). If the parent is missing from the source for that beat, ADD the parent to anchor the child — never remove the child. Always depict children wholesomely — awake, upright, clothed for the era, supervised; never `half asleep`/lying down/undressed/alone near a vehicle. **Children also stay clear of physical danger (fire, weapons, heights, water, moving vehicles) and keep a mild emotional range — see rule 12 and §F.** Reword §C filter-tripping phrasings so the child stays in frame. Genericize real brands/logos; flag a real celebrity lead name to the user and fall back to a non-celebrity tag only as a last resort. Full rules and rewrite recipes in `references/minor-and-policy-safety.md`.
12. **Visible emotion on every character (no lifeless faces)**: every character who appears in a scene prompt — the primary subject AND every secondary tagged character — must carry a **specific, readable emotion** that fits the block's emotional beat, expressed through **face + posture/gesture** (e.g. "with a tight, restrained anger, jaw set hard", "watching with quiet, aching hope", "looking away in guarded shame"). Never leave a character with only a neutral action and no feeling ("standing by the desk", "holding a folder", "looking at the door") — that produces a blank, soulless, mannequin face. Match each character's emotion to their role/temperament (the eye-"quality" on their sheet) and to what this specific moment makes them feel; two people in one frame usually feel **different** things. This applies to **scene prompts only** — the character **sheet** keeps its fixed `Calm neutral expression`, because it is a reference image, not a story beat. **Children are the one exception to the full emotion pool: keep a child's expression in a mild, age-appropriate sub-range (curiosity, shyness, quiet worry, delight, a hopeful/uncertain look) and never give a child a dark/intense descriptor (`terror`, `anguish`, `despair`, `grief`, `rage`, `trauma`) — put that intensity on a nearby adult instead. Children are also never placed near a physical hazard (fire, a weapon, a height, water, a moving vehicle); if the source implies danger, show the moment of safety instead.** Emotion vocabulary pool and worked guidance in `references/scene-prompt-formula.md`; full child tone/danger rules in `references/minor-and-policy-safety.md` §F.
13. **Footwear — characters are never barefoot unless the script says so**: the character **sheet** is a full-body image, so every sheet attribute clause MUST end with the lower garment + specific footwear (item 9 in `character-formula.md`) — e.g. "… with dark blue denim jeans and tan work boots". In **scene prompts**, include the lower garment + footwear in the scene garment whenever the shot shows the character full-body / head-to-toe (any wide or standing framing); upper-body-only framings (close-up, seated at a table, over-the-shoulder) need only the top garment. Match the shoes to the character's era, region, and profession, consistent with their sheet. Only render a character barefoot when the script explicitly places them so (beach, in bed) — and then write "barefoot" deliberately, never leave it unstated.

If the user supplies their own character sheet, still build the JSON scene-layout map (Step 2) but use their sheet as the canonical `characters[]` source instead of deriving descriptions yourself, and skip writing a new sheet in Step 3 — adopt their tags if present, otherwise assign tags (still enforce `@tag` + garment reuse, the tagged-once rule, self-contained settings, and count match).
